# Changelog

All notable changes are recorded here. The project follows semantic versioning; pre-1.0 the schema and APIs may still evolve between minor versions.

## Unreleased

- **cli/core/mission-control (R2)**: **Hermes-style workspace + per-project isolation** — multiple modernization projects now coexist with nothing shared. A `loom-workspace.yaml` manifest holds named projects; `resolveProjectContext` (threaded through `context.requireProfile`) picks the active one, with an explicit `--profile`/`--data-dir` always **short-circuiting to the legacy single-profile path** so every prior flow is untouched. Each project gets its own data dir (so `loom.db`/the atlases/`b-repo` can't collide), its own file-skills dir, and **namespaced tools** (`<project>__<name>`) so external/MCP tools can't clash; DB-scoped skills + memory were already per-project. `loom project new|list|use|current` manage the workspace, and Mission Control gains a project switcher (`GET /api/projects` + `?project=`). See [ADR 0006](docs/decisions/0006-workspace-project-isolation.md).
- **surveyor/conductor (R1)**: the **AI-explorer** + the **deep-map swarm** — the harness can take a huge, menu-driven app end to end. The explorer reaches the screens behind buttons/menus/tabs that link-following can't (e.g. a `qpmenu` shell): it enumerates a page's interactive controls, an LLM **chooser** (steered toward navigation, away from destructive actions) picks one, it's clicked, and the new state is deduped by `screenKey` — all hard-bounded; proven live in a browser. Crawl/explore output persists into a new **UI atlas** (`uiatlas.db`: states + forms + nav edges), wired into `loom crawl`. The **deep-map swarm** (`deepMap` + an LLM area-mapper) fans many same-instruction sub-agents over the worker pool to map an app too large for one context into a deduped picture (summary + entities + links), documentation-only. `captureFrames` captures iframe/frameset screens a single-document capture misses.
- **evaluator/conductor (R1)**: the deterministic judge is now the **full seven layers** — visual, structural DOM, computed-style (existing) plus **functional/validation** (every form field + rule must survive — `diffForms`), **accessibility** (axe-core regressions A→B — `diffA11y` + an in-browser capture), and **anti-cheat** (no rebuilt file byte-identical to a legacy asset — `findCopiedAssets`/`scanAssets`). `evaluateScreen` wires functional + a11y + anti-cheat as additional gates (each optional, so existing callers are unchanged), and a unified `gradeScreen` collapses every layer into one verdict. A run-finish **run-summary** reflection memory recaps each shift for the next.
- **cli (R0)**: a **Hermes-style startup identity** on bare `loom` — the brass `LOOM` splash (24-bit gradient, ASCII fallback) over the active model/provider, project, and SQLite backend. The README leads with a terminal-hero identity + a dark-mode-aware three-keys mark.
- **mission-control/cli**: the **inventory board** — Mission Control now shows everything the harness has, in one place. `inventory(db)` + `GET /api/inventory` assemble the built-in tools (write_file + the atlas/eval/crawl capability surface), the external MCP servers it consumes (from the profile), every skill it knows (tracked DB skills with tier/status/use+success counts, plus on-disk SKILL.md files), and a scan of the shared DIGIT/Copilot home (`~/.copilot`): its skills, agents, and MCP servers. The dashboard is redesigned as a premium **glassmorphism** surface with **light/dark** modes, the Loom three-keys mark, and Tools/Skills/MCP/DIGIT panels. Human-in-the-loop is now consistent everywhere: a shared `applyGateDecision` means approving a **skill gate from the browser** activates the skill (previously only the CLI did).
- **cli/scripts (L8)**: **pod packaging hardening**. `loom doctor` gains a **browser** check that actually launches Playwright Chromium (the surveyor + evaluator both need it; the pod has it cached) a **data-dir** check that fails if the resolved data dir lives inside a git clone (so bank data can never be accidentally committed — risk register #10), and a **proxy** check that reports the outbound-proxy posture + whether the LLM endpoint bypasses it via `NO_PROXY` (credentials redacted). A new `scripts/offline-rehearsal.sh` dry-runs the whole pod loop on blocked-internet — install (frozen/`--offline`) → build → tests → doctor → the full fixture MAP→…→EVAL pipeline → an optional `loom update` across two tags — failing loudly on the first broken step, with scratch data kept outside the clone. The exit-code + `--json` contract is documented as **frozen at v1.0.0**. POD-RUNBOOK gains an offline-rehearsal section.
- **core/conductor (L7)**: **webhook notifier** — the "ping a human" surface. `notifyWebhook` POSTs a Teams/Slack-compatible JSON payload (the chat app renders `text`); `runPipeline` fires it on **stop-the-line** and **shift-done** when `LOOM_WEBHOOK_URL` is set, best-effort and isolated (a down webhook never affects the run). Proven end-to-end against a local capture server.
- **mission-control/cli (L7)**: **Mission Control** — the local web dashboard. A new `@loom/mission-control` package: `dashboardState(db)` assembles the full read model (run + stage, screen tally by state, the gates/questions inbox, the span cost rollup, recent events); `startMissionControl({ db })` serves a self-contained, `@loom/tokens`-themed single-page dashboard (no build step, pod-friendly) that polls `/api/state` and writes back **only** human gate/question decisions (`POST /api/gates/:id`, `/api/questions/:id`) — the one documented exception to the conductor's single-writer rule. `loom ui [--port N] [--open]` launches it (read-only over `loom.db`, localhost-bound). A run is drivable gate-to-gate from the browser: approve a ship gate or answer a blocked screen's question and the inbox updates.
- **core/conductor (L7)**: **OpenTelemetry spans** — the observability spine for Live Now and cost views. A new `spans` table (migration 6) + `SpanStore` records OTel-GenAI-shaped spans (trace/parent ids, kind `llm|tool|eval|stage|attempt`, status, start/end, duration, free-form `gen_ai.*` attributes) correlated to run/WP/attempt; `startSpan`/`endSpan` track in-flight operations, `record` logs completed ones, and `aggregate(runId)` sums token usage + duration. The conductor records a GenAI span per build attempt (model + token usage). Read-only for the UI. **Optional OTLP export**: `toOtlpTraces`/`exportSpansOtlp` shape spans into an OpenTelemetry GenAI-convention OTLP/HTTP payload (stable id→hex, `gen_ai.*` attributes, unix-nano timing) and POST them to `<endpoint>/v1/traces`; `runPipeline` exports a run's spans at finish when `OTEL_EXPORTER_OTLP_ENDPOINT` is set — best-effort and isolated, so a down collector never affects the run.
- **cli (L7)**: **`loom watch`** — a single-glance terminal dashboard of the active run for SSH/pod use: the run + stage, a screen tally by state, token spend, the gates/questions waiting in the inbox, the recent event feed, and a **heartbeat-staleness "is it wedged?"** flag (stale past 6 min). The frame renderer (`renderWatchFrame`) is a pure function — snapshot-testable by feeding it fixed state — and `--json` returns the same resolved frame data. The browser Mission Control + OTel spans land next in L7.
- **agents/cartographer/cli (L6)**: **consensus / adversarial verification**. `judgePanel(gateway, { claim, evidence, judges?, quorum? })` runs N independent, skeptical judges over a claim + its evidence and returns a quorum verdict (strict majority by default; an unparseable judge counts as a reject), for the **subjective** calls the deterministic 7-layer evaluator can't make — recovered-doc accuracy, plan quality, ambiguous parity. First use: `verifyScreenDocs` / `loom atlas verify-docs` puts each recovered screen doc to the panel against the atlas facts it was generated from and flags ones the source doesn't support (exit 7 if any flagged) — catching a doc that reads fine but claims a control the legacy screen never had. The deterministic gates are untouched and remain the source of truth. (DoD: a subtly-wrong recovered doc is caught by the panel.)
- **agents/conductor (L5)**: **cache-stable context**. `buildSystemPrompt(role)` heads every agent's system prompt with a fixed `LOOM_IDENTITY` + `LOOM_SAFEGUARDS` preamble (the bootstrap identity/safeguards, injected once) then the role task — deterministic, no volatile content, so the cacheable prefix is byte-identical call to call and shared across roles; the Builder's system prompt is now a module-level constant. The agent loop also caps each tool result at `maxToolOutputChars` (Builder default 20k) so one runaway output can't blow the transcript. (DoD #2: the system-prompt prefix is byte-stable across turns.)
- **core/conductor (L5)**: **memory consolidation**. `MemoryStore.consolidate(project, { maxFacts? })` compacts project facts as the Reflector re-discovers them shift after shift — a loss-safe dedup (facts with the same normalized body collapse to the newest) plus an optional recency trim; worklog/reflection and other projects are untouched. `runPipeline` runs it (dedup only) at each run boundary and emits a `memory.consolidated` event. (DoD #1: shrinks a bloated fact set without losing key facts.)
- **skills/cli (L4)**: **DIGIT skill interop** — `loom skills export --target digit --out <dir>` writes the project's `SKILL.md` library in the layout DIGIT/Copilot reads (`<dir>/<name>/SKILL.md`), and `loom skills import --from <dir>` pulls external `SKILL.md` files into the project's `skills.dir`. Both run on `copySkillDir`, which re-serializes each file through the parser (normalized output, malformed files skipped) so the round-trip is format-faithful — our `SKILL.md` _is_ the agentskills.io/DIGIT shape. One-way-optional interop, never a runtime dependency (per [ADR 0004](docs/decisions/0004-self-contained.md)). Proven end-to-end: export → import yields byte-identical skills.
- **core/conductor/cli (L4)**: skills now **compound** — the self-improvement loop closes. Every skill recalled into a work order is credited with the screen's outcome: the conductor calls `SkillStore.recordUse(id, { success })` for each recalled skill on a WP pass/fail (`buildWorkOrder` now surfaces the recalled skill ids). `recordUse` applies the chosen **"human-approve, then auto-promote"** policy — a human still activates every draft at the skill gate, but thereafter an **`active`, `generated`** skill that reaches **N successful reuses** (`DEFAULT_PROMOTE_AFTER` = 3; `loom run --skill-promote-after <n>`) **auto-promotes to the `bundled` (global) tier** (`project` → null) and ranks higher in recall via its `success_count`. Guardrails are tested: a **draft** is never auto-promoted (the gate is never skipped), a **project**-tier skill is never promoted (no cross-client convention leak), and only successes count toward the threshold. Promotion emits a `skill.promoted` event for Mission Control. New concept doc: [Skills & memory](docs/concepts/skills-and-memory.md). This is the only place the harness "auto-approves" anything — and it only graduates an already-human-approved, already-proven skill.
- **rebrand → Loom Harness (L0)**: the product is now **Loom Harness**, driven by the `loom` CLI. The npm scope is `@loom/*`; the command is `loom`; the profile is `loom.config.yaml`, state is `loom.db`, and the data dir is `LOOM_DATA_DIR` — each with a documented fallback (`harness.config.yaml`, `harness.db`, `HARNESS_DATA_DIR`, `HARNESS_PROFILE`, `HARNESS_SQLITE_BACKEND`) so existing setups keep working. A new `@loom/tokens` package holds the brand design tokens — palette, semantic dark/light theme, `tokens.css`, and the woven-grid ASCII banner — shared by the CLI and the coming Mission Control WebUI; `loom --help` now prints the lockup. No behavior change; full suite green.
- **conductor/cli (M6)**: **shift-mode safeguards** — the start of unattended runs that never thrash. `runPipeline` takes run-level limits (`shift`): a cumulative **token budget**, a **wall-clock** cap, and **stop-the-line** (halt after N consecutive screen failures). Any trip stops the run gracefully (status `stopped`, a `shift.stopped` event, `stopReason` in the result), leaving the rest of the scope untouched rather than burning budget. `loom run --shift [--budget-tokens N] [--hours H] [--stop-after-failures N]` wires it up (`--shift` defaults stop-the-line to 3); the run output reports the stop reason. Per-attempt guards (iterations/tokens/wall-clock/no-progress) already bounded a single build; this bounds the whole run.
- **cli (M4)**: `loom crawl` — runs the surveyor against the running legacy app and prints a UI inventory (screen key, URL, link count). Form-login + start path + excludes come from a new `crawl` profile block (credentials via env vars, never the file). Verified end-to-end against the fixture (logs in as `analyst`, finds the list + wizard).
- **surveyor (M4)**: the crawler's core — `@loom/surveyor`. `domSignature` fingerprints a screen's structure (sibling subtrees collapsed, text ignored) so "same screen, different data" is one state; `screenKey` hashes path + frame + signature (origin and query dropped, so local↔prod and data variants like `/deal?id=1`/`id=2` collapse). `crawl` is a breadth-first walk that dedupes by state key and URL, records each state's same-origin links (`extractLinks`), and is hard-bounded by `maxStates`/`maxVisits` (no runaway). The `visit` seam keeps the BFS pure and tested without a browser. The **live crawler** `crawlApp` drives a real browser through a **persistent** session (`CrawlSession` — the login cookie carries across visits), does a **form-login** bootstrap, and walks the protected app; `exclude` keeps destructive links (e.g. `/logout`) out of the walk. Proven end-to-end: it logs into the bundled fixture as `analyst` and discovers the list + wizard (self-skips without a JDK/browser).
- **agents/cli**: **GitHub Copilot login is now a first-class, default provider.** `CopilotDriver` drives the authenticated `copilot` CLI headlessly (`copilot -p … --output-format json`) — **no `LLM_BASE_URL`/`LLM_API_KEY` required**, auth comes from the user's `copilot login` session, and the model is selectable. `loom init` auto-detects the Copilot CLI and defaults `driver: copilot` (no key in `.env`); `--driver` overrides; the OpenAI driver remains for environments with a direct BYOK key (e.g. the pod). `loom doctor` reports Copilot availability; `loom models list`/`test` now **show which provider is active and whether the model is yours to choose** (Copilot login) or locked (Azure/OpenAI key). Session-expiry is detected and surfaced as a re-auth hint for long shift-mode runs. Conformance-tested against a stubbed `copilot` binary (CI-safe, no live login). Corrects [ADR 0001](docs/decisions/0001-model-b-direct-llm.md): the harness still owns its loop (Model B), but the gateway transport is now Copilot-by-default.
- **conductor**: the Builder is now a pluggable `BuildStrategy`. With a Copilot login (no key) the conductor delegates the build to Copilot's own agent (`runCopilotAgent` → `copilot --allow-all-tools -C <b-repo>`, which writes the files), since you can't drive a tool loop through the completion API there; the deterministic 3-gate evaluator still gates the result, so Copilot's self-assessment is never trusted. `loom run`/`resume` pick the strategy from the driver automatically. So a Copilot-login developer can now run the **whole** pipeline — MAP → build → eval → fix — with no key. Proven end-to-end against a stubbed copilot agent.
- **evaluator (M5)**: the **structural/semantic DOM** layer — `diffDom` compares two normalized DOM trees and reports, with paths + reason codes, exactly what a pixel gate can't see (missing `<select>` option, `text`→`password` swap, relabelled control, dropped field, changed link target/role). The browser gains `captureDom` (extracts the normalized tree), and the conductor's EVAL now runs it as a **second gate** alongside visual — a pixel-perfect rebuild that drops a dropdown option is still blocked, and FIX feedback lists each structural diff. Calibrated both directions; the real-browser walking skeleton passes both gates.
- **evaluator (M5)**: the **coverage ledger** (`coverageLedger`) — the "no screen left behind" guarantee. Reconciles the MAP's static screen inventory, what the crawler reached, and what's been rebuilt; flags crawl gaps (planned-but-not-crawled), runtime-only screens (crawled-but-unplanned), and discovered-but-unbuilt screens, with a coverage %. `notBuilt` must be empty before a ship gate opens. Pure + deterministic.
- **evaluator (M5)**: the **computed-style** layer — `diffStyles` compares per-element style digests (typography, colour, borders, spacing; `captureDom` extracts them when asked) and runs as a **third gate** — catching sub-threshold "death by 1px" drift a font-size or border change that moves too few pixels to trip the visual gate. A pixel-perfect, structurally-identical rebuild with a font drift is still blocked.
- **cartographer (M3)**: the MAP stage deepens from the struts action graph to the full screen→layout→fragment→form picture. Dependency-free custom parsers for JSP (taglibs, forms+fields, nav links, iterations), Tiles (`tiles-defs.xml` layout composition), and `web.xml` (servlets/filters); `ingestLegacyWebapp` wires them into one idempotent CodeAtlas (tile_def/taglib/servlet/filter nodes; extends_tile/uses_taglib/includes/links_to/submits_to edges; parsed forms on each JSP node). New: the PageRank **repo-map** (`repoMap` — a compact whole-app overview naming every screen, well under an 8K-token budget), `discoverLegacyWebapp` (one struts path → the whole webapp's sources), and an enriched `sliceForScreen` (forms + taglibs). FTS5 full-text **search** over the graph (`atlas.search`, BM25-ranked over names/kinds/docs, LIKE fallback — FTS5 is present in the pod's node:sqlite backend). The **summarization docs pass** (`summarizeScreens`) recovers the missing documentation — one grounded LLM summary per screen, stored on the node, searchable, and shown in the repo-map. CLI: `loom map` auto-builds the enriched atlas; `loom atlas repomap`/`slice`/`find`/`summarize` read or enrich it.
- **conductor**: the Builder now builds with context. MAP builds the **enriched** atlas (Struts + Tiles + web.xml + JSPs, auto-discovered) and the work order (`buildWorkOrder`) carries the cartographer's full output — recovered documentation, parsed forms (fields + options), the **real legacy JSP source** to reproduce, and the whole-app repo-map — instead of just the screen name. (Verified: the work order reaching the Builder contains the action class, form fields, and JSP source.)
- **conductor**: the walking skeleton — `runPipeline` drives a screen MAP → CRAWL → PLAN → (BUILD → EVAL → FIX) over the durable task graph + event log, and **resumes** a killed run (interrupted attempts reconciled, only unfinished work packages re-processed). The Builder's `write_file` tool enforces protected paths (writes confined to the b-repo). The capture seam is injected so the whole state machine is tested deterministically (mock LLM, no browser); a self-skipping end-to-end test proves the fixture login goes MAP → build → eval → passed at < 2 % diff with a real browser.
- **cli**: pipeline + board verbs — `loom map` (browser-free MAP stage), `loom run` (full pipeline, exit 7 when a screen blocks), `loom resume` (latest interrupted run), and the read-only board `loom wp list`/`wp show`/`logs` over `loom.db`. `loom run` against the bundled fixture is verified end-to-end: one command, MAP to a passing screen.
- **core**: optional `source`/`app`/`target`/`eval` profile blocks (the pipeline's inputs); `TaskStore.latestRun({status})` for resume auto-detect.
- **profiles**: a committed `profiles/fixture` profile that drives the bundled legacy fixture.
- **evaluator**: the deterministic judge's foundation — `diffImages` (masked pixel diff, pure/no-browser), `scoreVisual` (per state×viewport verdict with human-approvable per-screen deviations), and `evaluateVisual` (diff+score a set of A/B captures); mutation-tested in both directions.
- **browser**: `@loom/browser` — a thin Playwright wrapper (`captureScreenshot`, `BrowserSession`, `canLaunchBrowser`) shared by the evaluator now and the surveyor later; browser-dependent tests self-skip where no browser is installed.
- **cli**: `loom eval --a <url> --b <url>` — captures both, diffs, scores, gates (exit 1 on fail), and can write diff images + `scorecard.json`. Works for legacy-vs-rebuilt or local-vs-production fidelity.
- **fixture**: the legacy test bed — a dependency-free Java app (`LegacyApp.java`, JDK-only) serving login/list/wizard/popup/frameset with sessions + 2000s markup, plus authentic `legacy-src/` Struts/JSP/Tiles source artifacts for the cartographer. `@loom/test-kit` `LegacyFixture` compiles + launches it for tests and the pipeline. Verified end-to-end: `loom eval` against the running fixture passes A-vs-A and fails A-vs-different.
- **docs**: documentation foundation — architecture overview, concept pages, guides, ADRs (0001–0005), per-package READMEs, CONTRIBUTING, and this changelog.

## v0.1.0 — M0 foundations (2026-06-15)

- **core**: SQLite adapter with a `node:sqlite` fallback (tested against both backends), forward-only migrations, append-only event log, profile/config loader with git-tree refusal.
- **agents**: `LlmGateway` with OpenAI-compatible (production) and Anthropic drivers; the guarded `AgentRunner` (max-iterations / token / wall-clock / no-progress); `resolveModelProfile`; the model-adaptive `ContextPacker` (budgets 128K–1M, tokenizer counting with fallback, shrink-ladder packing, vision degradation).
- **cli**: the registry-driven `harness` spine — `--json` contract and documented exit codes on every command, a `cli-conformance` test, and `doctor` / `status` / `update` / `db` / `profile` / `models` / `init`.
- **test-kit**: a scriptable mock OpenAI server.
- **CI**: lint/format/build/test on Ubuntu + Windows; secret scanning + a production-dependency license gate; tag-triggered releases.
- **Decision**: Model B — the harness owns the agent loop and calls the endpoint directly; nothing routes through external agent tooling.
